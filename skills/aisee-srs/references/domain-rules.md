# aisee:srs — 需求域识别规则

在确认门禁前，用本文件识别需求域和交付形态。

## 需求域类型

| 需求域 | 适用场景 |
|--------|----------|
| software | 通用软件系统、Web 应用、App、小程序、桌面应用、SaaS、管理后台 |
| backend | 后端服务能力、API 能力、集成服务、内部平台服务 |
| cli | 命令行工具、开发者工具、运维工具、本地处理工具 |
| job | 定时任务、异步任务、批处理、导入导出流水线、通知任务 |
| hardware | 实体产品、板卡、传感器、执行器、工业设备、仪器 |
| embedded | 固件、MCU/SoC、RTOS、驱动、Bootloader、BSP、设备运行时 |
| hybrid | 同一产品范围内同时包含软件服务和硬件 / 设备 / 固件能力 |

## 分域写作规则

- 始终描述用户可见、操作者可见、系统可见或设备可见的行为。
- 不写实现架构、API endpoint、数据库表、引脚表、寄存器表、RTOS 任务设计、驱动代码、BOM、PCB 布局或制造指令。
- 对软件和后端需求，记录业务对象、角色、状态、流程、约束和外部可观察的服务行为。
- 对 CLI 和 Job 需求，记录使用者、触发方式、输入来源、输出 / 结果、失败行为、幂等期望、可见性和业务目的。除非用户明确把具体命令参数或调度配置作为需求，否则不要写完整命令契约。
- 对硬件和嵌入式需求，记录设备能力、运行环境、输入 / 输出信号的业务含义、操作者流程、安全 / 质量约束、可观察状态、错误行为、校准 / 维护期望和验收方向。不要写硬件架构或固件设计。
- 对混合系统需求，在需求层面区分业务能力、软件侧行为、设备侧行为和跨边界交互。协议、资源分配和架构决策交给 `aisee:architecture`。

## 交付形态信号

根据交付形态决定后续交接提示：

| 信号 | 后续提示 |
|------|----------|
| 屏幕、页面、弹窗、表单、导航、角色可见 UI | 通常需要 UI Content |
| 组件库、视觉方向、Design Tokens、参考截图 | 通常需要 Design Spec |
| 技术栈未知、既有系统耦合、外部系统、异步处理、迁移、权限、设备运行时、硬件约束 | 通常需要 Architecture |
| API / service-only 行为 | change artifact 通常需要 service contract |
| CLI / 工具行为 | change artifact 通常需要 cli contract |
| 定时或异步行为 | change artifact 通常需要 job contract |
| 硬件 / 设备能力 | change artifact 可能需要 hardware / firmware / runtime / verification contract |

## 确认摘要要求

生成 SRS 前，确认摘要必须包含已识别的需求域和交付形态。如果需求域不清晰且会影响范围，先问一个聚焦问题；如果不明显影响范围，可以记录为 `[ASSUMPTION]`。
