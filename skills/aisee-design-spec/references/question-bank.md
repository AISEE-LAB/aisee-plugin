# aisee:design-spec — 问题库

在 Phase 1 输入门禁和后续澄清时读取。只选择真正影响设计规范的问题，不要全部询问。

---

## 追问优先级

1. 设计策略：adopt / extend / rewrite
2. 组件库 / 设计系统 / theme 来源
3. Logo、参考图、截图、Design Assets 来源
4. 目标平台与产品形态
5. UI Content 页面和流程范围
6. 品牌约束、信息密度和可访问性要求

每轮最多 3 个问题。超过 3 轮仍不明确时，把剩余问题写入 Open Questions。

---

## 设计策略

- 本次是直接采用现有组件库 / 设计系统，还是在组件库基础上扩展，还是完全重写视觉体系？
- 如果已有组件库，是否要求严格遵循默认设计，还是允许品牌化覆盖 token？
- 哪些部分必须保留现有视觉，哪些部分允许调整或重写？

---

## 组件库与主题

- 当前项目使用哪个 UI 组件库或设计系统？版本是否已确定？
- 是否已有主题 token、CSS 变量、Tailwind 配置、Figma variables 或设计系统文档？
- 哪些组件可以直接使用，哪些需要业务封装，哪些禁止自定义？
- 是否有既有页面作为组件使用方式的参考？

---

## 设计来源

- 是否有 Logo、品牌色、字体、参考图、截图、竞品图或设计资产索引？
- 哪些参考图只作为信息密度/布局参考，哪些作为品牌风格参考？
- 是否存在不能模仿的第三方品牌、Logo、文案或专有视觉元素？
- 是否需要把 design-assets 中的某些资产作为规范来源引用？

---

## 平台与产品形态

- 目标端是 PC Web、H5、App、微信小程序、桌面端，还是多端？
- 产品更偏管理后台、SaaS 工具、消费 App、营销页、内容产品、编辑器、数据分析、工作台，还是其他形态？
- 主要使用场景是高频工作、低频配置、内容浏览、交易转化、协作处理，还是数据分析？

---

## UI Content 与界面模式

- 哪些页面或流程必须被设计规范覆盖？
- 是否存在特殊界面模式，例如地图、日历、聊天、看板、画布、文件管理、直播、设备控制、数据驾驶舱？
- 哪些页面需要高信息密度，哪些页面需要更强品牌表达？

---

## 品牌与体验约束

- 设计需要偏稳重、专业、效率优先、年轻化、消费感、品牌展示，还是其他方向？
- 是否有合规、行业、无障碍、国际化或适老化要求？
- 动效、插画、图标和空状态是否需要统一风格？

---

## Open Question 处理

如果用户没有回答但不阻塞规范生成，记录为：

`[OPEN QUESTION] {问题} — 影响范围 {scope} — 建议确认对象 {owner/source}`

只有在设计策略、目标端或核心来源完全无法判断时才停止等待用户回答。
