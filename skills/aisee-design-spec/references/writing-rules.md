# aisee:design-spec — 写作规则

每次生成设计规范时读取。

## 通用规则

- 默认使用中文；除非用户明确要求其他语言。
- 删除所有未使用的模板占位段落，不要留下 `{placeholder}`。
- 如果某节确实无内容，写“无”，不要删除关键章节。
- Open Questions 不得为空白占位；没有问题时写“无”。
- 不要把假设写成事实；关键设计规则必须能追踪到来源、用户决策或明确假设。

## 边界规则

- 不生成图片、图标、插画、背景或素材。
- 不生成 SRS 或 UI Content，不重新定义业务流程、页面字段或用户操作。
- 不写具体页面像素稿、Figma 节点结构、CSS、组件 props、文件结构或实现任务。
- 不写 API endpoint、数据库字段或服务拆分。
- 不从参考图照抄第三方品牌、Logo、文案或专有视觉元素。

## 来源规则

- 优先引用 `design-assets` 中已有 Source ID；没有 ID 时使用 `DSRC-xxx`。
- 来源可信度必须标注 high / medium / low。
- `design-assets` 管资产事实，`design-spec` 只引用并归纳规则，不复制资产清单和生成参数。
- 参考图只能作为风格、密度、布局原则或交互线索，不等同于最终设计稿。

## 策略规则

- `adopt`：强调遵循现有组件库和 theme，少写自定义 token。
- `extend`：强调组件库基础上的 token 覆盖、业务封装和改造边界。
- `rewrite`：强调新的设计语言、token、组件原则和界面模式，但仍需来源或明确用户决策。
- 设计策略证据不足时标注 `[DESIGN-DECISION-REQUIRED]`。

## 界面模式规则

- Screen Patterns 必须从 UI Content 的页面、流程、角色、平台和信息密度中归纳。
- 不把页面模式固定为列表页、详情页、表单页等枚举。
- 每个 Pattern 必须写适用场景、内容组织、操作区、状态反馈、响应式变化和禁止事项。

## 组件规则

- 组件策略必须覆盖：直接使用、业务封装、自定义、禁止魔改。
- 如果技术上下文不能确认组件库，标注 `[TECH-CONTEXT-MISSING]`，不要替用户选型。
- 不写组件 props；只写使用边界、状态规则和体验约束。

## 下游提示规则

- 给 `aisee:design-assets` 的提示只说明如何根据规范生成参考图、素材或 visual brief。
- 给 `aisee:change-plan` 的提示只说明设计约束可能影响 change 边界的事实和原因，不直接拆 change。
- 给实现阶段的提示只写设计约束，不写代码。
