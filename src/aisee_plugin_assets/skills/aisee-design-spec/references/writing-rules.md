# aisee:design-spec — 写作规则

每次生成设计规范时读取。

## 通用规则

- 默认使用中文；除非用户明确要求其他语言。
- 删除所有未使用的模板占位段落，不要留下 `{placeholder}`。
- 如果某节确实无内容，写“无”，不要删除关键章节。
- Open Questions 不得为空白占位；没有问题时写“无”。
- 不要把假设写成事实；关键设计规则必须能追踪到来源、用户决策或明确假设。
- Design Read 必须具体到产品类型、目标受众、平台、气质、任务目标和禁用方向，不能只写“现代、简洁、高级”。

## 边界规则

- 不生成图片、图标、插画、背景或素材。
- 不生成 SRS 或 UI Content，不重新定义业务流程、页面字段或用户操作。
- 不写具体页面像素稿、Figma 节点结构、CSS、组件 props、文件结构或实现任务。
- 不写 API endpoint、数据库字段或服务拆分。
- 不从参考图照抄第三方品牌、Logo、文案或专有视觉元素。

## 来源规则

- 优先引用 `design-assets` 中已有 Source ID；没有 ID 时使用 `DSRC-xxx`。
- 来源可信度必须标注 high / medium / low。
- `design-assets` 管资产事实，`design-spec` 只引用并归纳规则，不复制资产清单和生成参数。
- 参考图只能作为风格、密度、布局原则或交互线索，不等同于最终设计稿。

## 策略规则

- `adopt`：强调遵循现有组件库和 theme，少写自定义 token。
- `extend`：强调组件库基础上的 token 覆盖、业务封装和改造边界。
- `rewrite`：强调新的设计语言、token、组件原则和界面模式，但仍需来源或明确用户决策。
- 设计策略证据不足时标注 `[DESIGN-DECISION-REQUIRED]`。

## Design Read 与参数规则

- `visual_density / content_density / brand_expression / motion_level / component_customization` 使用 1-5 档，不使用 1-10，避免伪精确。
- 参数必须影响后续章节：组件策略、Screen Patterns、tokens、动效、响应式和验收规则至少应引用其中的关键判断。
- 管理后台、工作台、CRM、ERP、运营工具默认优先任务效率和信息密度，不要套营销页式大 Hero 或强品牌视觉。
- 营销页、品牌页、作品集和消费级首屏可以提高 `brand_expression`，但仍必须保持文案可读、CTA 明确和移动端首屏可用。
- 动效规则只写范围和原则，不写具体实现代码或动画库 API。

## 界面模式规则

- Screen Patterns 必须从 UI Content 的页面、流程、角色、平台和信息密度中归纳。
- 不把页面模式固定为列表页、详情页、表单页等枚举。
- 每个 Pattern 必须写适用场景、内容组织、操作区、状态反馈、响应式变化和禁止事项。

## 组件规则

- 组件策略必须覆盖：直接使用、业务封装、自定义、禁止魔改。
- 如果技术架构不能确认组件库，标注 `[ARCHITECTURE-CONTEXT-MISSING]`，不要替用户选型。
- 不写组件 props；只写使用边界、状态规则和体验约束。
- 现有系统二开时，优先复用既有组件库、theme、tokens 和页面模式；不得因为参考图更好看就重造基础组件。
- 新项目才讨论组件库采用、扩展或重写策略；证据不足时写待确认，不替用户选择。

## 反模式规则

- Do / Don't 必须是项目相关反模式，不要只写通用口号。
- 常见应审查的 AI 默认反模式：卡片套卡片、三等分功能卡、无依据紫蓝渐变、按钮文字换行、导航桌面两行、placeholder 充当 label、只有成功态没有 loading/empty/error、移动端横向溢出、未确认的大面积动效。
- 不要把所有反模式都列入规范；只保留与当前产品、平台、组件库和参考来源有关的条目。

## 视觉验收规则

- 验收规则必须能被后续 `ui-contract.md`、`implementation-bridge` 或 `aisee:verify` 引用。
- 如果引用了 `dev-visual-brief` 或参考图，写清需要截图比对的页面、section、状态和重点。
- 如果没有视觉参考图，也要写基础验收：响应式、对比度、状态完整性、组件一致性。

## 下游提示规则

- 给 `aisee:design-assets` 的提示只说明如何根据规范生成参考图、素材或 visual brief。
- 给 `aisee:change-plan` 的提示只说明设计约束可能影响 change 边界的事实和原因，不直接拆 change。
- 给实现阶段的提示只写设计约束和验收重点，不写代码。
