# Image-first 视觉闭环

## 目标

当任务主要追求视觉质量时，先生成或读取清晰参考图，再提取视觉线索，最后输出下游 brief。不要从抽象审美规则直接跳到前端实现。

本文件只约束 `aisee:design-assets` 的视觉输入产出，不让本 skill 写代码。

## 触发条件

优先使用 image-first：

- 官网、营销页、活动页、作品集、产品落地页、App 首屏或 H5 首屏。
- 用户要求“高级感”“更好看”“重设计”“参考图”“视觉方向”“图片转页面视觉”。
- 用户上传截图、参考图、草图、品牌图或现有视觉资产，希望提取风格或延展新页面。
- change 后续需要引用视觉参考图、StyleSpec 草稿或 `dev-visual-brief` 做实现验收。

不强制使用 image-first：

- 后台表格、普通业务表单、权限配置、低视觉风险的内部工具。
- 已有明确 design-spec、组件库和 tokens，只需要小范围内容变更。
- 用户只要求 SRS、UI 内容规格、接口、数据模型或代码修复。

## 执行顺序

1. 读取 brief：平台、页面类型、受众、业务语气、现有品牌/组件库、禁用方向。
2. 判断是否需要生成新参考图、分析已有参考图，或两者都做。
3. 按页面、section、关键状态拆分参考图，不把长页面压成一张难以分析的大图。
4. 对每张参考图提取视觉语言、布局结构、组件形态、素材用法、可实现性和不清楚项。
5. 需要长期沉淀时输出 StyleSpec 草稿，并提示交给 `aisee:design-spec` 归纳。
6. 需要进入实现时输出 `dev-visual-brief`；它只描述视觉输入，不写业务逻辑或代码方案。

## 参考图粒度

- 单页面短流程：可以 1-2 张参考图。
- 长页面 / Landing：按 Hero、Proof、Feature、CTA、Footer 等 section 分图；默认 4-8 张，不生成一张超长图。
- App / 小程序 / H5：按页面或关键状态分图；列表、详情、表单、空状态、错误状态可拆开。
- Dashboard / 管理后台：优先生成关键区域或状态，不用满屏压缩导致字段不可读。
- 用户给一张模糊或压缩图时，先分析可见内容；细节不足时建议生成新的 section/detail 参考图，而不是猜。

## 分析清单

每张参考图至少记录：

- 目标页面 / section / 状态。
- 可读文字和不可读文字范围。
- 主视觉层级、视觉重心和信息密度。
- 色彩、字体气质、圆角、阴影、边框、分割线和材质。
- 布局网格、留白、卡片/区块节奏和响应式暗示。
- 组件形态：按钮、输入、导航、Tab、卡片、列表、弹窗、空状态。
- 素材路径或素材需求：背景、图片、插画、图标、装饰层、透明素材。
- 实现风险：文字不可控、Logo 漂移、组件库不匹配、移动端可能溢出、对比度风险。

## 禁止项

- 不把参考图当最终设计稿或规范事实源。
- 不直接生成前端代码。
- 不从图片中新增未确认业务需求、字段、权限或接口。
- 不机械描摹位图成为不可维护 UI。
- 不为了视觉效果重造已有组件库可以满足的基础组件。
- 不把整套 StyleSpec 复制进 change 内 `ui-contract.md`；change 只引用路径和本次适用约束。
