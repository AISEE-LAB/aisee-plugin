# aisee:ui-content — 写作规则

每次生成 UI 内容规格时读取。规则适用于标准模式和 Epic 模式。

## 通用规则

- 删除所有未使用的模板占位段落，不要留下 `{placeholder}`。
- 如果某节确实无内容，写“无”，不要删除关键章节。
- Open Questions 不得为空白占位；没有问题时写“无”。
- 不新增未被 SRS、用户或现有事实来源确认的业务需求。
- 不写颜色、字体、间距、栅格、布局稿、视觉风格、组件库选择或前端实现方案。
- 不写 API endpoint、数据库字段、服务拆分或代码文件结构。

## 场景规则

- `new-build`：可生成完整页面内容规格，所有待建页面可视为 New。
- `enhancement`：只展开 Changed / New / Deprecated；Existing 只引用来源，不重写完整规格。
- `inventory`：只整理 Existing / Unknown，不设计新需求。
- Unknown 必须写入 Open Questions，不要用猜测补齐。

## 页面规则

- 每个页面详情都必须保留入口来源、完成后去向、返回规则、参与流程和页面内流程语境。
- 页面局部流程放在页面详情内，不要只放到流程总览或 `shared-flows.md`。
- 页面锚点使用小写页面 ID，例如 `### PAGE-001` 可被链接为 `#page-001`。
- 单个页面详情控制在 2 屏 Markdown 内；复杂页面优先压缩表格描述，而不是拆掉上下文。

## 流程规则

- 流程总览只帮助读者理解链路，不重复页面字段、页面状态和页面内操作细节。
- `shared-flows.md` 只用于跨模块、跨端、跨角色或长链路流程。
- `shared-flows.md` 每一步必须引用 PAGE / MODAL / NAV / 非页面承载物 ID。

## Epic 规则

- Epic 模式索引文档只写总览，不展开完整页面详情。
- 模块文档可独立阅读，不要求读者先读索引才能理解页面内容。
- 单个模块文档最多包含 8 个页面详情；超过时继续按子模块拆分。
- 平台差异复杂时才生成 `platform-diff.md`。

## 平台差异规则

- 平台差异以内容、入口、操作、授权、反馈和能力限制为中心。
- 无差异时写“无差异”，不要留空。
- 不把平台差异写成视觉布局差异。

## 可选内容规则

- 埋点候选是可选章节；如果需求没有分析诉求，可以写“无”。
- 文案占位用于承接业务语义，不扩写品牌语调或视觉表达。
