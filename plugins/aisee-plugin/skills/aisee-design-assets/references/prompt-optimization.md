# 提示词优化

## 目标

在调用图片模型前，把用户意图整理成稳定、可执行、可复核的 prompt。不要做 Prompt Profile UI，不维护复杂版本系统，不替用户增加未确认业务需求。

## 默认策略

- skill 层默认优化提示词。
- 优化前先理解用户意图：判断用户是在要参考图、生成素材、素材识别、局部内容优化、保结构重绘、透明素材，还是 image-object handoff。
- CLI 层使用 `--augment` 做结构化增强；用户可用 `--no-augment` 关闭。
- 用户 prompt 已经精确时，只做格式化和约束补齐。
- 用户 prompt 很粗略时，可补充平台、尺寸、构图、视觉层级和避免项。
- 不要擅自添加品牌、业务流程、页面字段、人物、Logo、文案或具体数据。
- 只注入与当前任务直接相关的约束；不要把图标库、组件库、尺寸表、品牌保留、透明素材等规则一次性全部塞进 prompt。
- 约束优先写成短句，能用一句说清就不要展开为长列表。

## 最小注入原则

按任务类型选择必要约束：

- 文生参考图：平台、用途、画布比例、视觉目标、避免项。
- 图生图视觉变体：保留项、允许变化项和不应变动区域。
- 局部内容优化：用户意图、修改范围、保留项、允许变化、禁止项、是否需要 mask 或 image-object handoff。
- 来自 `aisee:image-object` 的局部编辑 handoff：只转写 image-object 提供的 source、mask、bbox、保护约束和目标路径。
- UI 图标相关：只说明使用标准图标占位和最终替换来源，不注入完整图标库列表。
- 透明素材：只说明独立主体、透明或可抠背景、无文字水印。
- StyleSpec 提取：不写生成约束，只要求结构化观察和不确定项标注。

如果某项不会影响当前输出，不写进 prompt，只记录到 brief 或 manifest。

## 生成前检查

每次生成前检查：

- 目标平台是否明确：mobile / mini-program / h5 / pc / desktop / multi
- 页面或素材用途是否明确
- 尺寸、比例或默认尺寸是否明确
- 主体、背景、构图和留白是否明确
- 视觉风格来源是否明确：用户描述 / 参考图 / StyleSpec
- 是否列出避免项：文字、Logo、水印、无关 UI、错误主体
- 是否说明透明输出需求
- 是否说明保留项和可修改项，尤其是图生图视觉变体或 image-object handoff
- 是否明确素材边界和文本策略，尤其是 banner、运营位、海报和电商图
- 局部优化是否有足够范围约束；没有 mask/bbox 时是否只能作为松散候选
- 输出文件路径是否在项目目录中

## Prompt 结构

使用以下结构；没有必要的项可以省略：

```text
任务类型：
目标平台：
输出用途：
画布尺寸：
视觉目标：
参考依据：
主体与内容：
构图与层级：
色彩与材质：
光影与氛围：
需要保留：
允许变化：
必须避免：
输出要求：
```

## 用户意图理解

先把用户请求归类，再写 prompt。不要把“优化一下”“更丰富”“更像参考图”直接转成泛化生图。

| 用户意图 | 需要确认/提取 | 默认处理 |
| --- | --- | --- |
| 生成参考图 | 平台、页面/场景、比例、视觉方向 | 文生图或图生图参考候选 |
| 规划素材 | 区域、视觉元素、UI/素材边界、文本策略 | 先输出 Asset Intent Scan |
| 生成素材 | 素材用途、尺寸、边界、透明、style_anchor | 单素材 prompt |
| 局部内容优化 | 源图、目标区域、要解决的问题、保留项、禁止项 | 受控编辑候选 |
| 保结构重绘 | 源图结构、模块语义、允许优化项 | structure-locked redraw |
| 精确对象处理 | mask、bbox、cutout、边缘或背景修补需求 | 交给 `aisee:image-object` |

局部优化请求必须拆成：

- `intent_summary`：用户真正想改善什么，例如“Hero 背景空、主体不突出、右侧插画细节不足”。
- `scope`：整图、section、banner 背景、局部区域、对象、mask/bbox。
- `preserve`：品牌、构图、文字区、产品比例、未授权区域。
- `modify`：只写本轮要改的视觉内容。
- `avoid`：文字、Logo、水印、无关主体、改变 UI 结构、破坏参考风格。
- `handoff_required`：是否需要 `aisee:image-object` 先提供 mask/bbox/cutout。

## 参考图 prompt 原则

- 参考图用于视觉方向，不要要求模型生成完整可交付 UI。
- 可以要求界面氛围、布局密度、信息层级、背景风格和视觉语言。
- 除非用户明确要求，不要让图片中出现可读文字。
- 避免自动生成品牌 Logo、水印、无关控件。
- 涉及通用图标时只使用标准图标占位；最终图标由图标库或组件库替换。

## StyleSpec 提取 prompt 原则

- 要求模型输出结构化规范，不只输出审美总结。
- 明确要求覆盖色彩、构图、图形语言、素材需求、禁用项、下游生成提示。
- 要求标注不确定项，不要把看不清的细节写成事实。

## 素材生成 prompt 原则

- 每个素材单独写 prompt，不把多个不同素材塞进同一张图。
- 明确素材用途、尺寸、透明需求、风格依据和避免项。
- 透明素材必须说明“独立主体、无背景、无文字、无水印”。
- 用于叠加的装饰层要避免投影、地面、复杂背景。
- 通用 UI 图标优先图标库，不走生图；只有品牌、业务专属或插画型图标才生成。

## 图生图视觉变体 prompt 原则

- 先写保留项，再写允许变化项。
- 不要要求模型重新设计整张图，除非用户确实要重做。
- 默认保持品牌元素、布局结构、视觉风格、色彩系统、字体层级、图标风格和不应变动区域。
- 涉及 Logo、品牌标识、真实产品图或文字密集 UI 时，不要求模型重画这些细节；优先保留原像素、确定性贴图、Figma 或前端实现。
- 输出新版本，不覆盖源图。

## 局部内容优化 prompt 原则

局部内容优化用于驱动 Image2 对参考图、banner、背景、插画或装饰层做受控完善，不用于对象提取、抠图、mask 生产、透明切图或背景修补执行。

- 不需要精确 mask 的松散优化可以留在本 skill，例如增强某个 banner 背景层次、补充插画细节、改善氛围光、扩大留白安全区。
- 需要像素级范围、对象边缘、局部抹除、背景修补、cutout、透明导出或用户指出“只改这个物体”时，先由 `aisee:image-object` 建立 workspace 和 mask/bbox。
- Prompt 必须先写“只修改指定范围/语义区域”，再写“保持其余区域不变”。
- Banner、运营位和 Web/App 参考图默认不生成文字；只保留可编辑文字安全区。
- 输出是候选图，不覆盖源图；结果需要在 manifest 或 index 中记录源图、scope、修改项和风险。

最小结构：

```text
任务类型：受控局部内容优化。
用户意图摘要：<用户真正想改善的问题>
输入图：<source/reference path>
局部范围：<自然语言区域；如有 mask/bbox/object id 必须写>
必须保持：<构图、品牌、文字区、主体比例、未授权区域>
本轮修改：<只写要新增、优化或修正的内容>
文本策略：<no-text | editable-text-zone | raster-text-confirmed>
必须避免：新增文字、Logo、水印、无关主体、改变 UI 结构、改动未授权区域。
输出要求：生成新候选，不覆盖源图；保持参考风格一致。
```

## Image-object handoff prompt 原则

- 局部编辑、背景修补、对象修边或从已有图片提取素材，先由 `aisee:image-object` 建立 workspace。
- 只使用 image-object 提供的 source、mask、bbox、object/cutout、保护约束和目标路径。
- 可以基于对话提炼“区域描述”和“优化意图”，但必须同时带上 image-object 的 mask / bbox / region / object 定位。
- 局部编辑时明确“只修改 mask / bbox / object 指定范围”。
- 不把 image-object workspace 的全部文件、缩略图或 `preview-cache/` 塞进 prompt。

## 优化前后记录

交付时可记录 prompt 摘要，不强制保存完整 prompt。需要追溯时在索引、manifest 或 brief 中保存：

- 原始用户意图摘要
- 优化后 prompt
- 避免项
- 来源参考图或 StyleSpec 路径

不得保存 API Key、完整敏感输入、私密文档原文。
